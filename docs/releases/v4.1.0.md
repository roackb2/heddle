# Heddle v4.1.0

Heddle v4.1.0 makes MCP-backed host extensions **self-contained**, so a host can
build a fresh conversation engine per request without paying MCP setup cost each
time. This is the seam a multi-tenant server needs: prepare an MCP server once,
then reuse the same prepared extension across many cheap, per-request engines —
each with its own API key and its own injected storage — with no per-engine MCP
prep and no per-request `.heddle/` directory.

This release is additive and backward compatible.

## Self-contained MCP host extensions

- **`prepareMcpHostExtension(...)` now embeds its resolved server config and
  cached catalog.** At runtime the extension resolves and executes its tools
  from that embedded data instead of re-reading the MCP config/catalog from the
  engine's `stateRoot`. One prepared extension can be shared across engines that
  point at different (or empty) state roots.
- **Why it matters.** Previously the extension read MCP state from
  `context.stateRoot` on every turn, so an embedder building a per-user,
  per-request engine had to write MCP config and refresh the catalog under each
  engine's state root. Now that cost is paid once at preparation time.
- **Execution stays stateless.** Each tool call spawns a fresh MCP subprocess
  from the embedded server config and closes it, and the same allow/deny tool
  policy is still enforced.

## Compatibility

- Fully backward compatible. A plain `defineMcpHostExtension(options)` with no
  embedded data keeps the original behavior: the toolkit reads the server and
  catalog from `stateRoot` at runtime (the local CLI-host path is unchanged).
- No public API removals or renames.

## Verification

- `yarn build`
- `yarn test`
- `npm pack --dry-run --cache /tmp/heddle-npm-cache`
- Verified end-to-end against a real MCP server: a prepared extension resolves
  and executes tools against an empty `stateRoot`, while a non-embedded
  extension resolves nothing there.
