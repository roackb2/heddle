# Heddle v3.5.0

Release date: 2026-06-08

Heddle v3.5.0 adds the first complete Model Context Protocol integration
foundation. Users can configure standard MCP servers, explicitly enable them
for a workspace, and expose their tool catalogs to future agent turns through
Heddle's normal approval-gated tool path.

## Highlights

### MCP servers can be configured with standard JSON

Heddle now reads a workspace MCP configuration file at `.heddle/mcp.json` using
the common `mcpServers` JSON shape used by other agent harnesses.

Why this matters:

- users can paste existing MCP server configuration snippets without learning a
  Heddle-specific format
- the web Settings -> MCP page can save, validate, and reload the same config
  file
- terminal users can open the config file with `/mcp config` instead of editing
  hidden state by hand

### MCP activation is explicit and workspace-scoped

Configured servers are not automatically exposed to the agent. Users can inspect
servers with `/mcp`, then enable, refresh, or disable them from terminal chat or
from the web MCP settings page.

Why this matters:

- a configured server and an activated server are separate states
- users remain responsible for which external MCP servers Heddle may connect to
  in a workspace
- enablement state stays in Heddle's workspace state while server definitions
  remain in the user-editable config file

### Agents can discover and call MCP tools

Enabled servers are refreshed into a cached MCP tool catalog, and future agent
turns can inspect available tools with `mcp_list_tools` and call them through
`mcp_call_tool`.

Why this matters:

- Heddle can use ecosystem tools such as Notion, Anytype, GitHub, and other MCP
  integrations without first building one-off toolkits for each service
- cached tool metadata keeps the runtime prompt compact while still giving the
  agent a standard way to discover external capabilities
- MCP tool calls still pass through Heddle's existing tool execution and
  approval boundaries

### MCP is documented as a core domain

This release adds MCP reference documentation, README coverage in English and
Chinese, slash-command tests, toolkit tests, control-plane routes, and a
service-level README for the MCP module.

Why this matters:

- contributors have a clear boundary for config parsing, activation state,
  catalog refresh, runtime tool injection, and client lifecycle behavior
- terminal and web surfaces share the same core MCP state
- future MCP work can focus on richer server transports, UX, and hosted
  workflows without changing the activation model

## Upgrade Notes

- No migration is required from v3.4.0.
- `heddle` and `heddle chat` continue to open the API-backed TUI.
- Add MCP servers in `.heddle/mcp.json` with an `mcpServers` object.
- Use `/mcp`, `/mcp config`, `/mcp enable <server>`, `/mcp refresh <server>`,
  and `/mcp disable <server>` in terminal chat.
- Use the web Settings -> MCP page to paste full MCP JSON, search configured
  servers, and enable or disable servers.
- Enabled MCP servers are available to future agent turns after refresh through
  `mcp_list_tools` and `mcp_call_tool`.
