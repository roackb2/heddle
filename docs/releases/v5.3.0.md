# Heddle v5.3.0

Heddle v5.3 lets a hosted product consume a short-lived OpenAI access token
without turning Heddle into an identity provider or credential store. It also
adds a stateless OpenAI device-login building block and a complete local JSON
persistence example for adopters who want the smallest durable starting point.

## Request-scoped model credentials

- **Hosts can provide one short-lived runtime credential for a turn.** The
  credential principal stays consistent across the main model request,
  compaction, and OpenAI-backed external-context tools.
- **Runtime access tokens are ephemeral by contract.** Heddle validates expiry
  and provider compatibility but never refreshes or persists the token.
- **Credential ambiguity fails explicitly.** A host cannot silently combine an
  API key with a request-scoped token or use a token for the wrong provider.
- **Stored OAuth remains a separate mode.** Existing refreshable credential
  stores keep their current behavior; the new runtime credential does not
  weaken that boundary.

## Hosted OpenAI device login

- **`OpenAiDeviceCodeAuthService` exposes stateless request and poll
  operations.** A host can present OpenAI's verification page and one-time code
  without proxying an OpenAI password or owning a callback allowlist.
- **Polling has explicit pending, expired, and authorized outcomes.** The final
  result contains only the access-token runtime credential needed by the
  engine.
- **Refresh material is discarded.** Heddle does not return, store, or refresh
  a device-login refresh token.
- **The host still owns product authentication and abuse controls.** Device
  challenges and runtime credentials should stay request- or browser-scoped,
  and hosted routes should be authenticated, rate-limited, and non-cacheable.

## Local persistence onboarding

- The Stage 06 persistence examples now present local JSON and PostgreSQL as
  sibling deployment postures.
- The local JSON verifier demonstrates fresh-engine conversation and archive
  recovery plus whole-root backup and restore.
- The example remains honest about its single-host boundary; multi-replica
  durability still requires host-provided remote repositories.

## Upgrade notes

- Install `@roackb2/heddle@5.3.0` to use request-scoped runtime credentials and
  `OpenAiDeviceCodeAuthService`.
- Existing API-key and stored-OAuth configurations remain compatible.
- The OpenAI device-code binding is an experimental provider-specific building
  block, not a promise that arbitrary third-party OpenAI OAuth clients or
  redirect URLs are supported.
- `@roackb2/heddle-remote@5.3.0` is released in lockstep. Its browser protocol
  is unchanged.
