# Heddle v3.7.0

Browser Automation now has workspace-level profile and display controls, making
the opt-in browser capability more practical for logged-in, user-supervised
web workflows.

## What's Changed

- Added Browser Automation settings in the control plane for enabling the
  capability, choosing the Heddle-managed profile, selecting headless/headed
  mode, and choosing the browser channel.
- Added slash commands for browser capability status, enable/disable, display
  mode, profile selection, channel selection, and manual profile-window login
  workflows.
- Added Heddle-owned browser profile windows so users can open the selected
  profile in headed mode, log in manually, and then reuse that persisted
  session for future browser runs.
- Improved browser snapshots so agents receive richer accessibility-tree
  context, raw DOM hrefs, bounded truncation metadata, and a larger snapshot
  budget for long account or listing pages.
- Hardened browser automation evidence and policy behavior by keeping snapshot
  refs scoped to current page state and avoiding site-specific URL inference.
- Fixed read-only postflight audit handling so read-only tool output paths do
  not get treated as filesystem mutations.

## Workflow Notes

Browser Automation remains opt-in. Enable it from Settings -> Browser
Automation or with `/browser enable`, then use headed mode or
`/browser open-profile` to prepare a logged-in Heddle-managed profile when a
site requires session state.

The browser tools still do not perform checkout, payment, booking, messaging,
account mutation, credential entry, or other irreversible actions. Those
actions remain outside the supported safety boundary.
