# Heddle v2.0.0

Release date: 2026-05-26

Heddle v2.0.0 makes the web-v2 control plane the default browser experience.
The packaged daemon, local build scripts, development commands, and CI browser
coverage now treat web-v2 as the supported UI, while the older web-v1 client
stays available only through explicit legacy commands.

## Highlights

### Web-v2 is now the default control plane

`heddle daemon` now serves the web-v2 browser client by default. The normal
`yarn build`, `yarn client:build`, `yarn client:dev`, and `yarn daemon:dev`
paths all point at `src/web-v2` and `dist/src/web-v2`.

Why this matters:

- new users see the current Sessions, Tasks, Settings, workspace switching,
  memory status, composer controls, and mobile layouts without choosing a
  special v2 path
- local development commands match the product surface that ships in the npm
  package
- the default daemon asset lookup no longer silently prefers the legacy web-v1
  bundle

### Web-v1 is explicit legacy fallback

The older browser client remains in the repository, but it is no longer the
default build or CI target. Use the explicit v1 scripts when you need to inspect
or serve it from a checkout:

```bash
yarn client:build:v1
yarn client:dev:v1
yarn daemon:dev:v1
```

Why this matters:

- v1 is still available for fallback investigation during the cutover
- default commands and documentation no longer imply that v1 is the supported
  browser surface
- future cleanup can remove v1 and unused backend logic from a clear boundary

### CI now validates the default browser experience

The browser integration job now runs the web-v2 suite by default and excludes
the web-v1 suite from normal PR checks. A new static daemon smoke verifies that
the built CLI can serve bundled web-v2 assets and load key routes without Vite.

Why this matters:

- CI follows the same UI that users get from the shipped daemon
- release verification covers both Vite-backed browser tests and bundled static
  daemon serving
- the package build path is less likely to drift from local frontend
  development

### Public docs now introduce the new web and mobile experience

The README and control-plane docs now describe web-v2 as the default browser UI,
remove legacy v1 screenshots from the main overview, and point users toward the
current Sessions, Tasks, Settings, workspace switching, memory status, composer,
and mobile workflows.

Why this matters:

- the project front page matches the product users actually see
- users get a direct path to try the new browser and mobile control plane with
  `heddle daemon`
- release notes and reference docs use the new consistent script names:
  `client:build:v1`, `client:dev:v1`, `client:build:v2`, and `client:dev:v2`
