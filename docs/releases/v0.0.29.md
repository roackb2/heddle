# Heddle v0.0.29

Release date: 2026-04-24

## Summary

Heddle v0.0.29 introduces agent-maintained workspace memory: Heddle can now learn durable project facts, workflow conventions, and operator preferences while it works, then recover that knowledge in later sessions through inspectable local catalogs.

This release also improves the surrounding runtime behavior so memory maintenance does not block normal chat, memory activity is visible from CLI and control-plane surfaces, and repeated discovery can turn into durable workspace knowledge instead of being lost after one session.

## Highlights

- Heddle can record durable memory candidates during normal agent work
- A maintainer path folds candidates into cataloged markdown under `.heddle/memory/`
- Future sessions can discover memory through explicit catalogs instead of opaque retrieval
- New `heddle memory` commands expose status, listing, reading, search, validation, and maintenance
- Interactive chat schedules memory maintenance in the background; one-shot ask mode can wait inline
- Memory reminders are fail-open instead of blocking final answers, preventing runaway token-burning loops when an agent does not satisfy a reminder
- The control plane and mobile shell show memory update activity so operators know when learning is happening

## What's New

### Agent-maintained memory

- Added catalog-backed workspace memory rooted at `.heddle/memory/`
- Added default catalogs for current state, workflows, preferences, domain knowledge, operations, relationships, and history
- Added candidate recording so the main agent can preserve reusable project knowledge without directly rewriting arbitrary notes
- Added maintainer tooling to update notes while preserving clear folder-level discovery paths
- Added maintenance logs and validation so memory changes can be audited

### Memory-aware agent runtime

- Added memory domain guidance to agent context so agents understand how Heddle-managed memory differs from repo docs or ad hoc notes
- Added `record_knowledge` and `memory_checkpoint` tools for durable learning decisions
- Changed post-mutation/final-answer memory handling to use fail-open reminders rather than blocking loops, so Heddle does not repeatedly spend user tokens trying to force a tool call before showing the final answer
- Hid raw internal tool transcript output from normal chat display so users see useful answers instead of implementation details
- Added background memory queue behavior for interactive chat and inline maintenance behavior for ask-style runs

### CLI and control-plane visibility

- Added `heddle memory status`, `list`, `read`, `search`, `validate`, and `maintain`
- Added repair/validation behavior for expected catalog shape
- Added control-plane memory service plumbing and UI indicators for memory updates
- Added memory state visibility to desktop and mobile session surfaces

### Reliability and safety polish

- Improved stale terminal chat lease handling so failed or abandoned runs do not trap a session indefinitely
- Preserved session mutation safety while allowing memory maintenance to avoid corrupting user-visible conversation state
- Added tests for catalog shape, maintainer behavior, memory integration, memory visibility, CLI behavior, and runtime memory hooks

### Documentation and discoverability

- Promoted knowledge persistence as a headline Heddle capability
- Added a concrete learning-loop example based on remembering a preferred ticket template
- Updated capabilities docs and README language for AI coding agent, persistent memory, local-first operation, and inspectable learning
- Added package metadata keywords for npm discovery around AI coding agents, agent memory, persistent memory, learning agents, and human-in-the-loop workflows

## Upgrade Notes

- Package version is now `0.0.29`
- Memory is stored locally under `.heddle/memory/`
- Existing projects can run `heddle memory status` or `heddle memory validate --repair` to initialize or inspect the catalog shape
- Memory maintenance is designed to be non-blocking in interactive chat, but users can inspect results with the memory CLI commands
- The memory system favors explicit markdown catalogs and auditability over opaque vector-only retrieval

## Known Limits

- Memory quality depends on real usage and agent judgment; the maintainer keeps discovery paths coherent, but users should still inspect important learned preferences
- This release provides the local memory foundation, not a hosted cross-repository knowledge service
- Memory catalogs are intentionally human-readable markdown, so very large or highly structured organizations may eventually want richer domain-specific loaders or storage backends
