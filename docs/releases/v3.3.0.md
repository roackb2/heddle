# Heddle v3.3.0

Release date: 2026-06-08

Heddle v3.3.0 is a feature release for long-running autonomous coding work. It
adds the first end-to-end Auto permission mode, gives agents a shared policy
envelope for declaring intended tool impact, and records richer approval-policy
evidence in traces so blocked runs are easier to diagnose.

## Highlights

### Auto mode can run trusted local work without routine approval

Heddle now has a user-facing permission mode model for Default, Auto, and
Custom. Auto uses a generated local-coding profile that lets normal trusted
work proceed under the current repo and explicitly trusted sibling repos while
keeping dangerous roots and production-like environments approval-gated or
denied.

Why this matters:

- users can switch into Auto from the web UI or TUI instead of approving every
  routine read, edit, verification, dependency, or git-stage action
- approval prompts for sibling repos can offer `Trust this repo`, which expands
  Auto's trusted root set without turning the workspace into Custom
- Custom remains reserved for a real hand-authored autopilot profile, so the UI
  does not claim custom autonomy when runtime behavior is still default approval

### Tool policy envelopes make agent intent explicit

Agent-capable tools can now carry a shared policy envelope that declares the
operation categories, intended roots, environment, expected effects, and
confidence for a call. The approval harness validates that envelope and combines
it with runtime facts instead of blindly trusting the agent's self-assessment.

Why this matters:

- shell and other flexible tools can still be useful in Auto mode without
  requiring the harness to perfectly parse every free-form command
- traces show both claimed intent and computed policy facts, which makes it
  easier to understand why an action was allowed, denied, or sent to approval
- tool authors have a single envelope shape and service README instead of
  inventing per-tool policy fields

### Autonomy policy is documented as a maintained domain

The autonomy approval service now has service-level documentation, focused
schemas, normalized profile handling, root-scope detection, and postflight audit
tracing.

Why this matters:

- future contributors have a clear boundary for permission-mode mapping,
  profile evaluation, trusted-root expansion, and trace semantics
- repo-root detection is bounded and uses project markers such as `.git`,
  `package.json`, `pyproject.toml`, `requirements.txt`, `Cargo.toml`, and
  `go.mod`
- the release keeps the mental model simple: Auto means normal local coding work
  can run in trusted repos; dangerous or unclear actions still stop

## Upgrade Notes

- No command migration is required from v3.2.2.
- `heddle` and `heddle chat` continue to open the API-backed TUI.
- Workspaces using Auto trusted sibling repos should use
  `autoTrustedRoots` in `.heddle/config.json`.
- `autoAllowedRoots` was part of the unreleased development branch only; the
  shipped config name is `autoTrustedRoots`.
- Custom permission mode is available only when `.heddle/config.json` contains
  a hand-authored `autopilot` profile with `mode: "autopilot"`.
