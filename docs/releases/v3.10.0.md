# Heddle v3.10.0

Heddle v3.10.0 broadens model provider support beyond the built-in OpenAI,
Anthropic, and Ollama paths. Users can now target multiple OpenAI-compatible
provider families with consistent model prefixes, shared model discovery, and
provider-specific smoke tests.

## What's Changed

- Added shared OpenAI-compatible provider profiles for LM Studio, LiteLLM,
  vLLM, Hugging Face router, OpenRouter, Together AI, and Groq.
- Added provider-owned model discovery so model pickers and model option APIs
  can surface available OpenAI-compatible models when the provider endpoint is
  reachable.
- Refactored Ollama onto the shared OpenAI-compatible adapter boundary while
  preserving Ollama-specific model listing through the local Ollama API.
- Added runtime credential handling for provider base URLs and API keys without
  hardcoding a single default local model.
- Added smoke test scripts for each OpenAI-compatible provider family, with
  opt-in behavior for providers that require local servers or external keys.
- Documented the supported provider families, model prefixes, setup variables,
  and local-model quality caveats in the README, zh-TW README, and provider
  reference docs.

## Workflow Notes

Use provider prefixes to choose the model family explicitly:

```bash
heddle --model ollama/llama3.2:latest ask "Summarize this repository"
heddle --model lmstudio/local-model ask "Summarize this repository"
heddle --model openrouter/meta-llama/llama-3.3-70b-instruct ask "Summarize this repository"
heddle --model groq/llama-3.3-70b-versatile ask "Summarize this repository"
```

For hosted OpenAI-compatible providers, configure the matching API key before
running Heddle. For local/self-hosted providers such as LM Studio, LiteLLM, and
vLLM, start the provider server first and configure the matching base URL if it
does not use the documented default.

Provider compatibility means Heddle can route requests and tool definitions to
that provider. It does not guarantee every model behind that provider is equally
good at coding-agent work. Smaller, older, heavily quantized, local, or
aggressively routed models may miss tool calls, ignore tool results, run slowly,
or answer confidently with incorrect repository facts.
