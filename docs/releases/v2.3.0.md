# Heddle v2.3.0

Release date: 2026-05-31

Heddle v2.3.0 makes active agent work easier to follow across the browser UI
and the API-backed terminal UI. This release focuses on live plan visibility,
clearer current/last activity status, reliable approval progression, and
keeping session welcome guidance out of recorded agent transcript state.

## Highlights

### Live agent plans are visible in web-v2 and cli-v2

When the agent creates or updates a plan, web-v2 and `chat-v2` now surface that
plan while the current turn is active. Plan facts are carried through the shared
control-plane session flow, while each interface owns its own rendering.

Why this matters:

- users can see what the agent is trying to do before the final answer arrives
- browser and terminal surfaces use the same plan state instead of separate UI
  assumptions
- mobile web keeps the plan compact with a collapsible panel

### Current and last activity status are clearer

The session activity surface now distinguishes the current thing the agent is
doing from the last completed activity. Web-v2 and `chat-v2` show active
thinking/tool/approval state near the prompt, with elapsed timing for active
work and a compact last-activity line for recent completed work.

Why this matters:

- users can tell the agent is still processing even when no tokens are streaming
- the UI avoids duplicating full approval or tool-call details in the status row
- the terminal prompt area now reads more consistently with the browser
  experience

### Approval progression is reliable after each decision

Control-plane approval handling now waits for the stored decision and refreshed
pending approval state before returning to the client. Web-v2 can reliably show
the next approval request after the user accepts or denies the current one.

Why this matters:

- users are not left with disabled approval controls while the agent is blocked
- the one-at-a-time approval model stays simple and predictable
- both web and terminal clients depend on the same control-plane state
  progression

### Welcome guidance is runtime context, not conversation history

Initial session guidance now comes from session runtime context rather than
recorded assistant messages. Interfaces decide how to render that guidance, and
the guidance disappears once the user starts the conversation.

Why this matters:

- the agent no longer sees UI welcome text as prior assistant transcript
- web-v2 can render a centered welcome panel for empty sessions
- cli-v2 can keep terminal-specific guidance without changing recorded session
  history
