# Heddle v5.2.0

Heddle v5.2 makes the path from one useful local conversation to a durable
hosted agent smaller and more explicit. New adopters can start with a structured
headless service, while production hosts can configure and assess the complete
conversation-persistence capability without hiding product-owned durability
decisions.

## Start with one structured conversation

- **`ConversationAgentService` is the smallest in-process SDK entry point.** It
  resolves shared runtime defaults, safely ensures one stable session, submits
  turns, and returns structured activities and results without owning a CLI,
  transport, database, or UI.
- **The engine remains available for progressive control.** Adopters can move
  from the headless service to `createConversationEngine` without changing the
  underlying session, turn, compaction, or persistence semantics.
- **SDK host responsibilities are clearer.** Headless and console onboarding
  services now share runtime policy while keeping structured product embedding
  separate from terminal evaluation and the full Heddle CLI products.

## One conversation-persistence capability

- **`persistence.conversations` groups the complete durable boundary.** Hosts
  configure the revisioned session repository and append-only archive repository
  together instead of reasoning about an incomplete persistence promise.
- **Readiness is inspectable without becoming a certification framework.**
  `ConversationPersistenceService.assess(...)` reports configuration gaps and
  the small set of host checks required for local or completed-conversation
  durability.
- **Existing integrations remain compatible.** The separate
  `sessionRepository` and `archiveRepository` inputs continue to work while new
  hosts adopt the capability-shaped configuration incrementally.

## Credential-safe compaction

- Automatic, final, manual, and direct-shell compaction now retain the active
  conversation's resolved credential principal.
- Browser-supplied API keys and selected OAuth credential stores are carried
  into summary generation instead of being re-resolved from a host environment
  or default auth store.
- Compaction keeps its purpose-specific model policy without silently switching
  whose credential is used.

## Durability guidance

- New architecture and programmatic guides inventory Heddle's durable state
  domains and distinguish refresh, process-replacement, and replica-replacement
  support levels.
- Completed-conversation durability is an explicit production boundary; it does
  not imply durable in-flight execution, SSE replay across replicas, or a
  universal storage abstraction.
- Hosts still own trusted tenant scope, schema and migrations, backup/restore,
  retention, deployment topology, and canonical product transactions.

## Upgrade notes

- Install `@roackb2/heddle@5.2.0` to use `ConversationAgentService`,
  `persistence.conversations`, readiness assessment, and credential-safe
  compaction.
- Existing engine repository inputs remain operational. Prefer
  `persistence.conversations` when adopting both durable conversation stores.
- Hosted or multi-user callers of `ConversationAgentService` must provide a
  stable session identity derived from trusted server-side scope.
- `@roackb2/heddle-remote@5.2.0` is released in lockstep. Its hosted run protocol
  is unchanged.
