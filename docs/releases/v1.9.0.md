# Heddle v1.9.0

Release date: 2026-05-22

Heddle v1.9.0 moves the new browser control plane from a static shell toward a
usable session workspace. The release adds web-v2 prompt submission, live
assistant streaming, session selection feedback, new-session creation, and
browser tool approvals while tightening the shared event and approval
boundaries underneath.

## Highlights

### Web-v2 can now run real session turns

The v2 session workspace now loads session detail through React Query and tRPC,
submits prompts to the control-plane API, renders streamed assistant output, and
keeps the selected session visible in the sidebar.

Why this matters:

- web-v2 is now useful for testing normal chat turns, not only route layout
- session switching and new-session creation are available in the v2 surface
- live status, loading, and error feedback are handled through the v2 API layer

### Tool approvals are available in web-v2

Pending tool approvals now render as a compact approval panel in the selected
session. The browser resolves approvals through the control-plane API using the
same core approval decision shape used by the chat engine.

Why this matters:

- approval UI reflects the runtime's actual approval state
- missed live approval notifications recover by polling the pending-approval API
  while a run is active
- denial remains the last action in the approval panel

### Live session events use shared conversation activities

The live event vocabulary now flows from the conversation engine boundary through
TUI, web-v1, and web-v2 consumers. web-v2 uses tRPC subscriptions for live
session updates instead of custom browser event plumbing.

Why this matters:

- streaming, tool status, approval, and compaction updates share one event model
- frontend callers consume a clearer API event shape
- docs now explain how live events propagate and where to extend them

### Approval logic is consolidated behind the approval service

Tool approval policy evaluation, human approval requests, and remembered project
rules now live behind the core approval service boundary. TUI and control-plane
hosts no longer reach directly into lower-level approval repositories for this
flow.

Why this matters:

- hosts call the approval service instead of duplicating approval behavior
- pending browser approvals are documented as server-held in-memory state
- contributors have a clearer README for approval storage and resolution flow
