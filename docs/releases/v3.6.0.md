# Heddle v3.6.0

Release date: 2026-06-09

Heddle v3.6.0 adds a workspace-scoped Browser Automation capability. Users can
turn it on explicitly, give future agent turns the right built-in skill
guidance, and expose the browser tools only when the workspace should allow
rendered page inspection or user-requested web workflows.

## Highlights

### Browser Automation is now an explicit capability

Browser Automation is off by default. Users can enable or disable it from the
web Settings -> Browser Automation page or from terminal chat:

```text
/browser
/browser enable
/browser disable
```

Why this matters:

- browser tools stay opt-in at the workspace level
- users can see whether the browser capability is enabled before relying on it
- Heddle can keep browser policy and profile behavior separated from ordinary
  tool availability

### Built-in skill guidance teaches when to use the browser

The release adds a Heddle-owned `browser-automation` skill that is packaged with
the runtime. When the capability is enabled, future agent turns see guidance for
when Browser Automation is appropriate, including rendered frontend
verification, visual inspection, product or listing research, and user-requested
website workflows.

Why this matters:

- the agent does not just see tools; it also gets usage guidance
- web search remains useful for discovery, while browser tools are framed as the
  way to inspect and operate rendered page state
- the built-in skill is reserved for Heddle's own browser capability, avoiding
  user or project skills shadowing the capability switch

### Browser tools are wired into runtime, TUI, and control plane

Enabling Browser Automation adds browser tools such as `browser_open`,
`browser_snapshot`, `browser_click`, `browser_screenshot`, and `browser_close`
to future default agent turns. The settings surface and slash commands share the
same core capability service.

Why this matters:

- terminal and web users manage the same workspace-scoped state
- runtime tool assembly remains centralized instead of duplicating browser
  enablement logic per host
- browser safety policy remains owned by the browser/tooling domain

### README and user docs were reorganized

The English and Traditional Chinese READMEs now use a clearer structure for what
Heddle can do, how to use it, and where major features fit. Agent Skills,
Browser Automation, and MCP coverage are easier to find from the top-level docs.

Why this matters:

- new users can understand Heddle's practical workflows more quickly
- Traditional Chinese docs now follow the same structure and depth as the
  English README
- feature documentation is less scattered across incremental release-era edits

## Upgrade Notes

- No migration is required from v3.5.0.
- Browser Automation remains disabled by default.
- Enable it from Settings -> Browser Automation or with `/browser enable`.
- Logged-in sites still require a browser profile with a valid session. The
  capability switch does not bypass browser policy or profile configuration.
- The browser automation skill is built into Heddle and is copied into the build
  output during `yarn build`.
